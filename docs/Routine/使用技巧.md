# Harness Engineering Routine 使用技巧

## 一、核心理念

Routine = **可重复的结构化工作流** + **多 Agent 协作** + **自动评估反馈**

```
输入 → Agent A → 评估 → Agent B → 追问 → 循环 → 报告
```

---

## 二、四种使用模式

### 模式 1：基础面试（对话式）

最简单的用法，一个 Agent 扮演面试官：

```yaml
name: basic-interview
steps:
  - role: system
    content: |
      你是Go面试官。每次只问一个问题。
      等待回答后再继续。会根据回答追问。
```

**技巧**：
- 用 `每次只问一个问题` 约束 Agent 行为
- 用 `等待用户回答` 防止自问自答
- 用 `逐步加深` 控制难度曲线

### 模式 2：压力面试（追问式）

加入追问机制，不接受模糊回答：

```yaml
steps:
  - role: system
    content: |
      面试规则：
      - 必须连续追问，直到问穿
      - 不接受模糊回答
      - 候选人说"差不多"→ 必须追问细节
      - 回答正确 → 提升难度
      - 回答错误 → 指出问题 + 继续追问
```

**技巧**：
- 用 `问穿` 关键词触发深度追问
- 用 `不接受模糊回答` 提升压力感
- 用 `指出问题` 让 Agent 纠正错误

### 模式 3：自动评分（反馈式）

每轮回答后自动评分：

```yaml
steps:
  - role: system
    content: |
      每轮回答后输出：
      【评价】
      - 优点：
      - 不足：
      - 建议：
      
      面试结束后输出：
      【最终评估】
      - 技术评级（初级/中级/高级）
      - 是否通过面试
```

**技巧**：
- 用 `【评价】` 格式化输出
- 用 `技术评级` 量化结果
- 用 `是否通过` 给出明确结论

### 模式 4：多 Agent 协作（完整式）

多个 Agent 各司其职：

```yaml
agents:
  interviewer:
    role: system
    prompt: 提出技术问题
    
  evaluator:
    role: system
    prompt: 评估回答质量
    
  followup_generator:
    role: system
    prompt: 生成追问
    
  knowledge_gap_analyzer:
    role: system
    prompt: 分析知识盲区

workflow:
  - step: ask
    agent: interviewer
  - step: evaluate
    agent: evaluator
  - step: followup
    agent: followup_generator
  - step: loop
    until: max_rounds
  - step: report
    agent: knowledge_gap_analyzer
```

## 三、Prompt 工程技巧

### 1. 角色定义

```yaml
# ❌ 不好的写法
content: "你是面试官"

# ✅ 好的写法
content: |
  你是一位一线大厂资深Go面试官（偏底层原理 + 高并发架构）。
  面试风格：专业、直接、不要过度鼓励。
```

**技巧**：
- 明确角色背景（一线大厂、资深）
- 明确技术方向（底层原理、高并发）
- 明确风格（专业、直接）

### 2. 规则约束

```yaml
# ❌ 不好的写法
content: "问一些Go问题"

# ✅ 好的写法
content: |
  面试规则：
  1. 每次只问一个问题
  2. 必须根据回答"追问"
  3. 如果回答模糊 → 追问细节
  4. 如果回答错误 → 指出问题并继续追问
  5. 如果回答很好 → 加大难度
```

**技巧**：
- 用编号列表明确规则
- 用 `如果...则...` 条件句
- 用 `必须` 强制约束

### 3. 难度控制

```yaml
content: |
  难度控制：
  - junior：基础概念
  - mid：源码 + 原理 + 场景
  - senior：架构设计 + trade-off
```

**技巧**：
- 用 `{{level}}` 变量动态调整
- 明确每个级别的考察重点
- senior 级别考察 trade-off 思维

### 4. 项目深挖

```yaml
content: |
  候选人项目：高并发API网关（Go + Redis + Etcd + Kafka）
  
  必问：
  - 如何做限流？（令牌桶/漏桶/滑动窗口）
  - goroutine 如何控制数量？
  - Redis 在这里解决什么问题？
  - Kafka 是否有堆积？如何处理？
  - Etcd 用来做什么？
  
  强制追问：
  - 如果流量突增 10 倍怎么办？
  - 单机 CPU 打满如何排查？
  - 如何避免 goroutine 泄漏？
```

**技巧**：
- 从简历中提取项目信息
- 列出必问问题清单
- 添加 `强制追问` 压力测试

### 5. 输出格式

```yaml
content: |
  【输出格式必须严格】
  【面试官】
  问题内容
  
  （等待用户回答，绝对不能自问自答）
```

**技巧**：
- 用 `【面试官】` 标记问题
- 用 `（等待用户回答）` 防止自答
- 用 `绝对不能` 强调约束

## 四、工作流设计技巧

### 1. 线性流程

```yaml
workflow:
  - step: intro
    agent: interviewer
  - step: question
    agent: interviewer
  - step: evaluate
    agent: evaluator
  - step: final
    agent: analyzer
```

适合：简单面试、代码审查

### 2. 循环流程

```yaml
workflow:
  - step: loop
    agent: interviewer
    until: max_rounds_reached
    max_iter: 10
```

**技巧**：
- 用 `until` 定义终止条件
- 用 `max_iter` 防止无限循环
- 终止条件可以是：`max_rounds_reached`、`candidate_pass`、`candidate_fail`

### 3. 条件分支

```yaml
workflow:
  - step: check
    condition: "score >= 80"
    next: hard_question
  - step: hard_question
    agent: interviewer
  - step: easy_question
    agent: interviewer
```

**技巧**：
- 用 `condition` 实现分支
- 用 `next` 指定跳转目标

## 五、评分系统技巧

### 1. 四维评分

```yaml
score:
  correctness: 1-10  # 正确性
  depth: 1-10        # 深度
  clarity: 1-10      # 清晰度
  practical: 1-10    # 实用性
```

**技巧**：
- 正确性：答案是否正确
- 深度：是否理解底层原理
- 清晰度：表达是否清楚
- 实用性：是否有实际经验

### 2. 反馈格式

```yaml
strengths:
  - 理解深入
  - 表达清晰
  
weaknesses:
  - 基础不够扎实
  
missing_points:
  - 没有提到写屏障
```

**技巧**：
- 优点要具体
- 不足要可改进
- 遗漏点要可学习

### 3. 评级标准

```yaml
level: junior/mid/senior

junior: 平均分 < 5
mid: 平均分 5-7
senior: 平均分 > 7
```

## 六、高级技巧

### 1. 动态 Prompt

使用变量让 Prompt 动态变化：

```yaml
content: |
  面试目标：评估候选人的 {{focus}} 能力
  
  难度控制：
  - {{level}}：基础概念
```

**技巧**：
- 用 `{{focus}}` 注入考察方向
- 用 `{{level}}` 注入难度级别
- 从 input 中读取变量值

### 2. 上下文传递

```yaml
input:
  resume: |
    我做过一个高并发API网关...
  
  focus:
    - GMP scheduler
    - goroutine
    - concurrency
```

**技巧**：
- 用 `resume` 传入简历
- 用 `focus` 传入考察重点
- Agent 会根据上下文生成问题

### 3. 追问策略

```yaml
# 基于评分追问
if score.depth < 5:
  ask("请更详细地解释")

# 基于薄弱点追问
if len(score.missing) > 0:
  ask(f"你似乎遗漏了 {score.missing[0]}")

# 提升难度追问
if score.total > 80:
  ask("请从源码层面解释")
```

### 4. 压力控制

```yaml
# 轻度压力
content: "请继续深入解释"

# 中度压力
content: "你的回答有一些不准确的地方"

# 高度压力
content: "不接受模糊回答，请给出具体细节"
```

## 八、使用流程

### 1. 准备阶段

```bash
# 1. 准备简历
vim resume.txt

# 2. 配置面试
vim interview.yaml

# 3. 设置焦点
focus: [GMP, GC, concurrency]
```

### 2. 执行阶段

```go
// 1. 创建 Routine
manager := NewRoutineManager(config)
instance, _ := manager.CreateFromConfig(ctx, "interview.yaml", params)

// 2. 启动面试
manager.Start(ctx, instance.ID)

// 3. 获取问题
question, _ := manager.GetQuestion(ctx, instance.ID)

// 4. 提交回答
manager.SubmitAnswer(ctx, instance.ID, "我的回答...")

// 5. 循环直到结束
```

### 3. 评估阶段

```go
// 1. 获取文本报告
report, _ := manager.GetReport(ctx, instance.ID)

// 2. 获取 Markdown 报告
mdReport, _ := manager.GetMarkdownReport(ctx, instance.ID)

// 3. 获取 HTML 报告
htmlReport, _ := manager.GetHTMLReport(ctx, instance.ID)
```

## 九、最佳实践

### 1. Prompt 设计

- ✅ 明确角色背景
- ✅ 列出具体规则
- ✅ 定义输出格式
- ❌ 模糊的描述
- ❌ 没有约束的指令

### 2. 工作流设计

- ✅ 每步职责单一
- ✅ 有明确终止条件
- ✅ 有错误处理
- ❌ 过于复杂的流程
- ❌ 没有终止条件的循环

### 3. 评分设计

- ✅ 多维度评分
- ✅ 具体的反馈
- ✅ 可执行的建议
- ❌ 单一分数
- ❌ 模糊的评价

### 4. 迭代优化

- ✅ 根据测试结果调整 Prompt
- ✅ 收集反馈改进流程
- ✅ 定期更新题库
- ❌ 一成不变的模板

## 十、常见问题

### Q1: Agent 自问自答怎么办？

**A**: 在 Prompt 中加入：
```
（等待用户回答，绝对不能自问自答）
```

### Q2: 评分不准确怎么办？

**A**: 
1. 细化评分标准
2. 增加评估维度
3. 使用 LLM 提升评估质量

### Q3: 追问太浅怎么办？

**A**: 在 Prompt 中加入：
```
必须连续追问，直到问穿
不接受模糊回答
```

### Q4: 循环停不下来怎么办？

**A**: 
1. 设置 `max_iter`
2. 设置明确的 `until` 条件
3. 添加超时机制

---

## 总结

Routine 的核心价值：
1. **结构化** - 把面试变成可重复的流程
2. **自动化** - Agent 自动提问、评估、追问
3. **量化** - 评分系统量化能力
4. **可迭代** - 根据反馈持续优化
