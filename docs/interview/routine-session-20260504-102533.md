**问题 1（基础-中等）**

在 Go 中，以下代码的输出是什么？为什么？

```go
func main() {
    s := []int{1, 2, 3, 4, 5}
    s1 := s[1:3]
    s2 := s[2:4]
    s1 = append(s1, 100)
    s2[0] = 200
    fmt.Println(s)
}
```


---

## 📝 第 1 轮 — LLM 生成答案

### 详细完整答案

**输出：`[1 2 200 100 5]`**

**核心原理：切片共享底层数组 + append 在容量充足时不扩容**

逐步分析：

```go
s := []int{1, 2, 3, 4, 5}
// 底层数组: [1, 2, 3, 4, 5]  len=5  cap=5

s1 := s[1:3]
// s1: [2, 3]  len=2  cap=4
// 指向底层数组索引 [1, 2]，cap=4 意味着可以向后扩展到索引 4

s2 := s[2:4]
// s2: [3, 4]  len=2  cap=3
// 指向底层数组索引 [2, 3]，cap=3 意味着可以向后扩展到索引 4
```

此时底层数组与三个切片的映射关系：
```
底层: [1,  2,  3,  4,  5]
       ↑   ↑   ↑   ↑   ↑
索引:  0   1   2   3   4
      s[0] s[1] s[2] s[3] s[4]
           s1[0] s1[1]  ← s1 len=2, cap=4
                 s2[0] s2[1]  ← s2 len=2, cap=3
```

```go
s1 = append(s1, 100)
// s1 的 len=2, cap=4，容量充足，不会分配新数组
// 直接在底层数组索引 2+2=3 的位置写入 100
// 底层数组变为: [1, 2, 3, 100, 5]
// s1: [2, 3, 100]  len=3
// s2: [3, 100]     因为 s2[1] 就是底层索引 3
```

```
底层: [1,  2,  3, 100, 5]
       ↑   ↑   ↑   ↑   ↑
索引:  0   1   2   3   4
           s1[0] s1[1] s1[2]
                 s2[0] s2[1]  ← s2[1] 被 append 覆盖为 100
```

```go
s2[0] = 200
// s2[0] 对应底层数组索引 2，将其改为 200
// 底层数组变为: [1, 2, 200, 100, 5]
```

```
底层: [1,  2, 200,100, 5]
       ↑   ↑   ↑   ↑   ↑
索引:  0   1   2   3   4
      s[0] s[1] s[2] s[3] s[4]
           s1[0] s1[1] s1[2]
                 s2[0] s2[1]
```

最终 `fmt.Println(s)` 打印 `[1 2 200 100 5]`。

**边界情况与陷阱：**

1. **如果 append 触发扩容**：若 `s1 = append(s1, 100, 200, 300)`（超过 cap=4），则 s1 会分配新底层数组，后续对 s1 的修改不再影响 s 和 s2。这是最常见的"切片陷阱"——append 是否影响原切片取决于是否扩容。

2. **切片重叠区域**：s1 和 s2 存在重叠（底层索引 2，即 s1[1] 和 s2[0]），修改一处会影响另一处。生产代码中应避免创建存在写入重叠的多个子切片。

3. **s 的 len 仍然是 5**：虽然底层数组被修改，但 s 的 len 没有变，`fmt.Println(s)` 仍然打印 5 个元素。

**最佳实践：**

- 需要独立副本时，使用 `copy(dst, src)` 或 `s1 := append([]int{}, s[1:3]...)`
- 不确定是否会扩容时，用 `s1 = s1[:len(s1):len(s1)]` 限制容量，让后续 append 强制分配新数组
- 多个 goroutine 并发写共享底层数组的不同子切片是数据竞争，需要用 channel 或 mutex 同步

---

### 一句话总结

**输出 `[1 2 200 100 5]`**：子切片 s1 和 s2 与 s 共享底层数组，s1 的 append 在容量内直接覆盖了 s[3]（即 s2[1]），s2[0]=200 又覆盖了 s[2]，最终反映到 s 上。

---

### 口头简答

输出是 `[1 2 200 100 5]`。因为 Go 的切片底层是指向同一个数组的，s1 从 s 里切出来容量还剩 4，append 一个 100 不会触发扩容，直接写到了底层数组索引 3 的位置，正好把 s2 的第二个元素覆盖成了 100。然后 s2[0]=200 又改了底层索引 2，最终 s 的值也被跟着改了。整个过程中三个切片都在操作同一块内存。

## 🎤 面试官提问

**问题 3（中等）**

以下 Go 代码存在什么问题？能否正常编译并运行？

```go
func main() {
    ch := make(chan int)
    ch <- 42
    fmt.Println(<-ch)
}
```


---

## 📝 第 2 轮 — LLM 生成答案

### 详细完整答案

**问题核心：无缓冲通道的死锁**

这段代码可以正常编译（语法完全正确），但运行时会触发 **fatal error: all goroutines are asleep - deadlock!**，发生死锁 panic。

**原理分析**

`make(chan int)` 创建的是**无缓冲通道**（unbuffered channel）。无缓冲通道的特性是：
- **发送操作 `ch <- 42` 会阻塞**，直到有另一个 goroutine 准备好接收该值
- **接收操作 `<-ch` 会阻塞**，直到有另一个 goroutine 发送了值

在这个例子中，整个程序只有一个主 goroutine。执行到 `ch <- 42` 时，当前 goroutine 被阻塞等待接收方；但接收方 `<-ch` 在同一 goroutine 中排在后面，永远执行不到。Go 运行时检测到所有 goroutine 都在休眠，触发死锁 panic。

**正确写法：**

```go
// 方案1：使用带缓冲的通道
func main() {
    ch := make(chan int, 1) // 缓冲大小为1
    ch <- 42                // 不阻塞，缓冲区有空位
    fmt.Println(<-ch)       // 从缓冲区取出
}

// 方案2：发送和接收放在不同 goroutine 中
func main() {
    ch := make(chan int)
    go func() {
        ch <- 42
    }()
    fmt.Println(<-ch)
}

// 方案3：先有接收者等待，再发送（同样需要 goroutine）
func main() {
    ch := make(chan int)
    go func() {
        fmt.Println(<-ch)
    }()
    ch <- 42
}
```

**边界情况与最佳实践：**

1. **编译期检测**：Go 编译器只能检测语法错误，死锁是运行时行为，编译期无法发现
2. **`go vet`**：标准 `go vet` 也检测不到这种死锁；需要 `go run -race` 或专门的静态分析工具（如 `staticcheck`）
3. **缓冲通道不是万能药**：缓冲大小为 1 只能容纳 1 个值，连续发送第 2 个值依然会阻塞
4. **select + default**：如果不想阻塞，可以用 `select` 加 `default` 分支做非阻塞发送
5. **设计原则**：发送和接收操作如果发生在同一个 goroutine 且顺序上发送在前、接收在后，无缓冲通道必然死锁

---

### 一句话总结

无缓冲通道的发送操作会阻塞直到有接收方，同一 goroutine 内先发后收必然死锁。

---

### 口头简答

这段代码能编译通过，但一跑就挂。因为 `make(chan int)` 创建的是无缓冲通道，往里面发数据会一直阻塞直到有人接收。但是代码里发送和接收都写在同一个 goroutine 里，发送完等接收，接收又等发送，两边互相等，Go 运行时直接报死锁 panic。改成 `make(chan int, 1)` 加个缓冲或者把发送扔到另一个 goroutine 里就行。
