## 🎤 面试官提问

Go 中的 `slice` 底层结构是怎样的？当向 `slice` 追加元素触发扩容时，底层数组的扩容策略是什么（扩容后的容量如何计算）？


---

## 📝 第 1 轮 — LLM 生成答案

### 详细完整答案

Go 中 `slice` 的底层结构定义在 `runtime/slice.go`：

```go
type slice struct {
    array unsafe.Pointer // 指向底层数组的指针
    len   int            // 当前元素个数
    cap   int            // 底层数组容量
}
```

**核心要点**：slice 本身是一个 24 字节（64 位系统）的描述符，并不存储实际数据。多个 slice 可以共享同一个底层数组。

---

#### 扩容策略（Go 1.18+）

扩容发生在 `append` 且 `len == cap` 时，核心逻辑在 `runtime.growslice`：

```go
// 伪代码，展示核心算法
func growslice(oldCap, newLen int) int {
    newcap := oldCap
    doublecap := oldCap + oldCap

    // 情况1: 所需长度超过 2 倍旧容量 → 直接用 newLen
    if newLen > doublecap {
        newcap = newLen
    } else {
        const threshold = 256
        // 情况2: 旧容量 < 256 → 直接翻倍
        if oldCap < threshold {
            newcap = doublecap
        } else {
            // 情况3: 旧容量 >= 256 → 平滑过渡公式
            // 从 ~2x 逐渐收敛到 ~1.25x
            for 0 < newcap && newcap < newLen {
                newcap += (newcap + 3*threshold + 4) / 4
            }
        }
    }

    // 最后会根据元素类型，按内存分配 size class 向上取整
    // 所以实际容量可能比计算结果更大
    return roundupsize(newcap * elemSize) / elemSize
}
```

**扩容策略三步总结：**

| 条件 | 策略 |
|---|---|
| `newLen > 2 * oldCap` | 直接用 `newLen` |
| `oldCap < 256` | 翻倍（2x） |
| `oldCap >= 256` | 公式 `newcap += (newcap + 3*256 + 4) / 4`，从 ~2x 逐步收敛到 ~1.25x |

**关键变化（Go 1.18 vs 之前）：**
- 旧版本：阈值是 1024，超过后固定 1.25x 增长
- 1.18+：阈值降到 256，且用平滑公式替代固定倍数，避免小 slice 增长过快、大 slice 增长过慢

#### 代码演示

```go
package main

import "fmt"

func main() {
    s := make([]int, 0)
    for i := 0; i < 10; i++ {
        s = append(s, i)
        fmt.Printf("len=%d cap=%d\n", len(s), cap(s))
    }
}
```

典型输出（64 位系统）：
```

len=4 cap=4
len=5 cap=8
len=6 cap=8
len=7 cap=8
len=8 cap=8

```

#### 边界情况与最佳实践

**1. 扩容触发复制**

扩容时底层会分配新数组，拷贝所有元素。原数组如果没有其他 slice 引用，会被 GC 回收。

```go
a := []int{1, 2, 3}
b := a          // b 和 a 共享底层数组
a = append(a, 4) // 可能触发扩容，a 指向新数组，b 仍指向旧数组
```

**2. 子 slice 与内存泄漏**

```go
func getFirst(data []byte) []byte {
    return data[:1] // 只用了 1 字节，但底层数组可能有 1GB！
}
// 正确做法：复制
func getFirstSafe(data []byte) []byte {
    result := make([]byte, 1)
    copy(result, data[:1])
    return result
}
```

**3. 预分配容量**

已知元素数量时用 `make([]T, 0, cap)` 避免多次扩容：

```go
// 差：多次扩容，频繁内存分配和拷贝
var s []int
for i := 0; i < 10000; i++ {
    s = append(s, i)
}

// 好：一次分配
s := make([]int, 0, 10000)
for i := 0; i < 10000; i++ {
    s = append(s, i)
}
```

**4. `len` vs `cap`**

`len` 是你能访问的安全边界，`cap` 是底层数组的物理边界。通过 `s = s[:cap(s)]` 可以暴露底层数组的全部已分配空间（后续 `append` 可能复用）。

---

### 一句话总结
Slice 底层是 `{指针, len, cap}` 三元组；扩容时 Go 1.18+ 对小 slice（<256）翻倍、对大 slice 用平滑公式从 ~2x 收敛到 ~1.25x，最后还会按内存分配 size class 向上取整。

### 口头简答
Slice 底层就是一个结构体，包含指向数组的指针、长度和容量三个字段。当 append 发现长度等于容量没位置了，就会扩容——小 slice 直接翻倍，大 slice 增长幅度逐渐收窄到 1.25 倍左右。扩容后底层数组会重新分配并拷贝，所以如果有其他 slice 共享同一个底层数组，它们之间就脱离关系了。

## 🎤 面试官提问

Go 中 `defer` 的执行顺序和参数求值时机是怎样的？如果一个函数中有多个 `defer`，它们的执行顺序是什么？`defer` 语句中的函数参数是在 `defer` 声明时求值还是在执行时求值？请举例说明。


---

## 📝 第 2 轮 — LLM 生成答案

### 详细完整答案

**1. 多个 defer 的执行顺序：后进先出（LIFO / 栈）**

`defer` 语句在函数返回前执行，多个 `defer` 按声明的**逆序**执行——就像叠盘子，最后叠的最先拿。

```go
func demo() {
    defer fmt.Println("1")
    defer fmt.Println("2")
    defer fmt.Println("3")
    fmt.Println("body")
}
// 输出:
// body
// 3
// 2
// 1
```

底层原因：每个 goroutine 有一个 `_defer` 链表，新 `defer` 插入链表头部（头插法），执行时从头部遍历，自然形成 LIFO。

**2. 参数求值时机：声明时求值，而非执行时**

`defer` 语句中的函数参数在 **`defer` 声明时就立即求值**，而不是等到实际执行时才求值。这是最常见的陷阱。

```go
func paramEval() {
    x := 1
    defer fmt.Println("defer:", x) // x 在这里被求值为 1
    x = 2
    fmt.Println("body:", x)
    // 输出:
    // body: 2
    // defer: 1    ← 注意是 1，不是 2
}
```

**但是**，如果通过闭包捕获变量，则读取的是执行时的值：

```go
func closureTrap() {
    x := 1
    defer func() { fmt.Println("defer:", x) }() // 闭包捕获变量引用
    x = 2
    fmt.Println("body:", x)
    // 输出:
    // body: 2
    // defer: 2    ← 闭包读到的是修改后的值
}
```

**3. 经典面试题：返回值修改**

这是区分参数求值和闭包的关键场景——命名返回值可以被 `defer` 修改：

```go
// defer 无法修改匿名返回值（return 时已将值拷贝）
func f1() int {
    x := 1
    defer func() { x++ }()
    return x  // 返回 1，defer 修改的是局部变量 x
}

// defer 可以修改命名返回值（return 时值已赋给返回值变量）
func f2() (x int) {
    defer func() { x++ }()
    return 1  // 返回 2，defer 修改的是返回值变量 x
}
```

**4. defer 与 return 的执行顺序**

`return` 不是原子操作，可以分解为：
1. 将返回值赋值给返回值变量
2. 执行所有 `defer`（此时可修改命名返回值）
3. 真正的函数返回（RET 指令）

```go
func order() (result int) {
    defer func() {
        result *= 2  // 步骤 2：result 现在是 4
    }()
    return 2  // 步骤 1：result = 2，步骤 3：返回 4
}
// 返回 4
```

**5. defer + recover 的 panic 处理**

`defer` 最常见的实际用途之一：

```go
func safeCall() {
    defer func() {
        if r := recover(); r != nil {
            fmt.Println("recovered:", r)
        }
    }()
    panic("boom") // panic 后所有 defer 仍会执行
}
// 只有 defer 中的 recover 能捕获 panic，且必须在发生 panic 的 goroutine 内
```

**6. 边界情况与最佳实践**

- **`os.Exit` 不会触发 defer**：`os.Exit(1)` 直接终止进程，所有 `defer` 都不会执行
- **循环中 defer 会堆积**：循环中 `defer` 直到函数返回才执行，可能导致资源泄漏。应使用 `func()` 包装或手动关闭
  ```go
  // 错误：所有文件在函数结束时才关闭
  for _, name := range files {
      f, _ := os.Open(name)
      defer f.Close()
  }
  // 正确：每次迭代结束时关闭
  for _, name := range files {
      func() {
          f, _ := os.Open(name)
          defer f.Close()
          // 使用 f
      }()
  }
  ```
- **defer 有微小性能开销**：纳秒级，绝大多数场景可忽略，但极热路径可考虑手动调用
- **资源管理用 defer 的惯用法**：锁、文件、连接等获取后立即 `defer Unlock()`/`defer Close()`

### 一句话总结
多个 `defer` 按 LIFO（后进先出）顺序执行，且 `defer` 语句中的参数在声明时立即求值（而非执行时），但闭包捕获的变量引用的则是执行时的值。

### 口头简答
多个 `defer` 就像一个栈，最后写的先执行，先写的后执行。参数的话，你在 `defer` 那一行传进去的值，当时就"定死了"，后面变量怎么改都不影响它。但如果你用闭包的方式，在里面引用外面的变量，那它读到的是真正执行那一刻的最新值。这个区别面试经常考，记住用 `fmt.Println(x)` 传参和用 `func() { fmt.Println(x) }()` 闭包，结果不一样。

---

## 📝 第 3 轮 — LLM 生成答案

### 详细完整答案

**GMP 模型核心概念**

| 组件 | 全称 | 角色 |
|------|------|------|
| **G** | Goroutine | 用户态轻量级协程，包含栈、指令指针、状态等。创建一个 G 只需几 KB 内存 |
| **M** | Machine/OS Thread | 操作系统线程，G 运行在 M 之上执行。Go 默认最多 10000 个 M |
| **P** | Processor | 逻辑处理器，是 G 和 M 之间的调度中介。`GOMAXPROCS` 决定 P 的数量（默认 CPU 核数） |

三个队列：
- **全局运行队列**：存放所有就绪的 G，需要加锁访问
- **本地运行队列**：每个 P 拥有一个 lock-free 队列（容量 256），存放该 P 私有的就绪 G
- **waitq 等待队列**：阻塞在 channel、网络 I/O 等上的 G 队列

**系统调用阻塞时的调度流程（关键考点）**

当一个 G 执行系统调用（如 `syscall.Read`）发生阻塞时：

**Step 1：M 与 P 解绑**
```
// runtime/proc.go 核心逻辑（简化）
func entersyscall() {
    // G 状态切换为 Gsyscall
    casgstatus(gp, _Grunning, _Gsyscall)
    
    // M 的 P 引用设为 nil，P 状态变为 Psyscall
    pp := _g_.m.p.ptr()
    _g_.m.p = 0
    atomic.Store(&pp.status, _Psyscall)
}
```

**Step 2：P 的处理 — 寻找新 M 接管**
- P 进入 `Psyscall` 状态，启动一个 **sysmon 监控计时器**
- 如果 sysmon 检测到该 P 处于 `Psyscall` 状态超过 **20µs**（短阻塞）且有空闲 M 或有新 G 等待，则：
  - 将 P 从原 M 剥离，状态改为 `Pidle`
  - 放入空闲 P 列表
  - 唤醒或新建一个 M 来接管这个 P 及其本地队列中的 G
- 如果 sysmon 检测到超过 **10ms**（长阻塞），直接剥离 P：
  - `handoffp()` 函数将 P 转交给其他 M
  - P 状态：`Psyscall` → `Pidle` → `Prunning`（被新 M 获取后）
  
```
// handoffp 核心：系统调用过长时解绑 P
func handoffp(pp *p) {
    // 如果 P 本地队列有 G 就绪，立即交给新 M
    if !runqempty(pp) {
        startm(pp, false)
    }
}
```

**Step 3：M 的处理 — M 阻塞在系统调用，不占用 P**
- M 带着当前 G 进入内核态执行系统调用，**P 已被剥离开**
- 当系统调用完成后（`exitsyscall`）：
  - **优先快速路径**：尝试重新获取原来的 P（如果 P 还在 `Psyscall` 状态），直接恢复执行
  - **如果 P 已被拿走**：将 G 放入全局运行队列，M 进入休眠或复用
  - **如果获取到空闲 P**：G 绑定到新 P 上继续执行

```
func exitsyscall() {
    oldp := _g_.m.oldp.ptr()
    // 快速路径：尝试拿回原来的 P
    if exitsyscallfast(oldp) {
        casgstatus(gp, _Gsyscall, _Grunning)
        return
    }
    // 慢速路径：放入全局队列，M 休眠
    exitsyscall0(gp)
}
```

**完整流转图**

```
   G 调用 syscall
        │
   M.P → nil          (解绑)
   P 状态 → Psyscall
        │
   ┌─ sysmon 检测 ──────────────────────┐
   │                                     │
   <20µs (短阻塞)                  >10ms (长阻塞)
   保留 P                          强制 handoffp
   M 等待返回                       P → 新 M 接管
        │                              │
   └─ syscall 返回 ────────────────────┘
        │
   exitsyscall:
   ① 尝试拿回原 P (P 还在 Psyscall) → 成功,恢复执行
   ② 原 P 已被拿走 → G 放全局队列,M 休眠
   ③ 获取到其他空闲 P → 绑定新 P,继续执行
```

**Work Stealing 机制**

当 P 的本地运行队列为空时，P 会按以下顺序窃取 G：

1. **从全局队列偷**（每 61 次调度连偷一次，其余时间按比例偷）：`globrunqget()`
   ```
   // 计算偷取数量：min(len(全局队列)/GOMAXPROCS, len(本地队列)/2)
   n = min(sched.runqsize/max(gomaxprocs,1), max(int32(len(local), 128)))
   ```

2. **从其他 P 的本地队列偷一半**：`runqsteal()`
   - 随机选取一个 P（避免偏向）
   - 偷取其本地队列的 **后半部分**（接近头部的最新 G，缓存局部性好）
   - 尝试 **4 次**，如果都失败则放弃
   
3. **再次检查全局队列**：因为 steal 过程中可能有 M 把 G 放入全局队列

4. **检查 netpoller（网络就绪的 G）**：`netpoll()`

5. 都没有 → P 进入自旋检查或最终空闲休眠

**关键设计要点**

- **handoffp 确保 P 不浪费**：Go 1.14 之前的 M 解除绑定时会阻塞在 syscall 导致 P 空闲，之后的实现确保 P 能给到其他 M
- **work stealing 是无锁的**：每个 P 的本地 runq 是无锁环形数组，steal 只是原子操作
- **sysmon 是独立 M**：不以 P 为中介，作为守护线程独立运行，避免被窃取 P 影响
- **netpoller 避免阻塞**：网络 I/O 走异步模型 + epoll/kqueue，G 进入 netpoll wait 而不是系统调用阻塞，不占用 M

---

### 一句话总结

G（goroutine）是用户态协程，M（machine）是 OS 线程，P（processor）是逻辑调度器；goroutine 进入系统调用时 M 与 P 解绑（handoff），P 被 sysmon 监控后转交给其他 M 防止资源浪费，而 work stealing 使得空闲 P 能从全局队列或其他 P 的本地队列中窃取 G 来执行，保证多核利用率。

---

### 口头简答

GMP 调度模型里，G 就是 goroutine 协程，M 是操作系统线程，P 是逻辑处理器，负责把 G 调度到 M 上执行。当一个 goroutine 执行系统调用阻塞时，调度器会把 P 从当前 M 身上剥离下来转交给其他空闲的 M，这样 P 本地队列里排队的 goroutine 就不会被卡住；等系统调用返回后，G 要么拿回原来的 P，要么进全局队列等下次被调度。work stealing 就是当某个 P 把自己的活干完了，会去全局队列或者其他 P 的本地队列里"偷"一半 goroutine 过来执行，保证所有核心都有活干，不会出现一个 P 累死、其他 P 闲着的情况。
