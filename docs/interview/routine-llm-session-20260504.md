## 🎤 第 1 轮

### 面试官提问

Go 中 `slice` 作为函数参数传递时，在函数内部修改元素会影响外部，但 `append` 追加元素却不一定。请解释：什么时候 `append` 会影响外部，什么时候不会，底层原因是什么？

### 候选人回答

slice底层是一个三字段结构体：ptr指向底层数组、len是当前长度、cap是容量。append时先检查cap是否足够，够则直接在底层数组追加并更新len；不够则触发扩容——分配更大的新数组，拷贝旧元素，追加新元素。扩容策略：cap<256时翻倍，>=256时约1.25倍增长。子切片与原slice共享底层数组，可能导致数据意外修改。

**遗漏:**

- 未区分 `s[i] = x`（总影响外部）vs `append`（len 不同步，外部不可见）
- 无代码示例说明两种场景

## 🎤 第 2 轮

### 面试官提问 (追问)

在 Go 中，以下代码有什么问题？如何修复？

```go
func process(items []int) []int {
    results := make([]int, 0)
    for _, item := range items {
        go func() {
            results = append(results, item*2)
        }()
    }
    return results
}
```

### 面试官提问 (新问题)

Go 中一个类型要满足某个接口，不需要显式声明 `implements`。请说明这种"隐式接口"设计带来的三个实际工程优势，并给出一个具体代码场景说明它如何影响你的架构决策。

### 候选人回答

Mutex是互斥锁，读写都互斥。RWMutex允许读并行，写独占。Go 1.18引入了饥饿模式防止长时间等待。sync.Map适合读多写少场景，内部用read-only map+dirty map实现，通过原子操作提升读性能。channel底层是hchan结构体，包含环形缓冲区、发送/接收队列和互斥锁。

## 🎤 第 3 轮

### 面试官提问

Go 中一个类型要满足某个接口，不需要显式声明 `implements`。请说明这种"隐式接口"设计带来的三个实际工程优势，并给出一个具体代码场景说明它如何影响你的架构决策。

### 候选人回答

defer按LIFO顺序执行。关键点：参数在声明时求值而非执行时；defer可以修改命名返回值；循环中的defer可能导致资源泄漏，应用匿名函数包裹。panic时defer仍会执行，recover只在defer中有效。Go 1.21引入了defer的性能优化，小函数内联。

### LLM 评估

| 维度 | 评分 | 说明 |
|------|------|------|
| 正确性 | 0/10 | 答非所问。问题问的是隐式接口设计，回答的是 defer 执行机制 |
| 深度 | 2/10 | defer 知识点覆盖较广，但与题目无关 |
| 清晰度 | 5/10 | defer 表述清晰有结构，但完全错位 |
| 实用性 | 0/10 | 对接口设计没有任何指导价值 |

**综合分: 14/100**

**优点:**
- defer 的知识点覆盖较全（LIFO、参数求值时机、命名返回值、panic/recover、性能优化）

**不足:**
- 完全答错题，疑似混淆了不同题目
- 没有任何关于接口设计的讨论

**遗漏:**
1. slice 本质是值类型（struct{ptr, len, cap}），传参时复制的是这个 header
2. 修改元素影响外部——因为 header 中的 ptr 指向同一底层数组
3. `append` 在 len < cap 时——底层数组不变，但调用方的 len 没变，所以"看不到"新元素
4. `append` 在 len == cap 时——分配新数组，header 全变，调用方不受影响
5. 使 `append` 影响外部的两种方式——返回新 slice 或传 `*[]T` 指针

**结论:** 候选人需要重新回答本题。
